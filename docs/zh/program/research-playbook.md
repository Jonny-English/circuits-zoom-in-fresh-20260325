# Research Playbook

当一篇文章或 notebook 开始显得过于开放时，就用这份 playbook 把工作拉回到可重复的研究循环里。

## 核心循环

这个仓库里的每个项目都应该经过同一套顺序：

1. 说清问题
2. 说清假设
3. 说清 baseline
4. 跑最小有用实验
5. 记录改了什么
6. 把 observation 和 interpretation 分开
7. 决定下一步实验

如果这 7 步里缺了一步，工作通常很快就会变得含糊。

## 读论文协议

每篇论文都做五遍阅读：

1. 摘要和引言：它真正想解决什么问题？
2. 图像扫描：最强的一张图是哪张？它承载了什么 claim？
3. 方法扫描：真正起作用的 intervention、representation 或 training move 是什么？
4. 证据扫描：如果论文是错的，哪条证据应该先崩？
5. 局限扫描：论文明确没有证明什么？

你的 reading note 不应该像逐段摘要，它应该像一个帮助决策的文档。

## 复现协议

在复现 notebook 时：

- 先跑 baseline
- 记录 seed、关键超参数和环境
- 每次只改一个变量家族
- 保留最小的一张图或一张表，用来对比 baseline 和 variant
- 失败要第一时间记下来，不要先在脑子里把它合理化

复现的目的，不只是“跑出同一张图”，而是学会知道哪些改动真的重要。

## Ablation 梯子

如果你不知道下一步改什么，就按这条梯子往下走：

1. 改 scale：强度、稀疏度、温度、系数
2. 改 capacity：feature 数、hidden dimension、数据量
3. 改 noise 或 corruption
4. 改 metric 或 evaluation target
5. 改任务本身

初学者最常犯的错，是过早跳到第 5 层。先待在最小有用的一层。

## 证据纪律

每份写作都要明确区分这三类内容：

- Observation：run 或 artifact 直接展示了什么
- Inference：对 observation 最保守的解释
- Speculation：更大的故事，但还需要新证据

这是让你听起来像研究者、而不是像爱好者的最快方法之一。

## Failure Analysis 提问框架

当结果看起来不强时，依次问：

- baseline 本身是不是就很弱？
- 我是不是改了不止一个变量家族？
- metric 和 claim 是对齐的吗？
- 效果是真有，但只是太小吗？
- artifact 是可读的，但其实还不足以定论吗？
- 什么证据会让我撤回当前这套解释？

把答案写下来。failure analysis 是技能，不是情绪。

## 每周复盘仪式

每周末回答这 4 个问题：

1. 我看到的最强证据是什么？
2. 最大的不确定性在哪里？
3. 什么失败了，或者最让我意外？
4. 下周做什么最小动作能让我少一点困惑？

答案保持简短。重点是保留判断，不是写日记。

## 沟通规则

- 写短 memo，不写长摘要
- 先给判断，再补细节
- 能先给证据就先给证据
- 风险要显式写出来
- 结尾给一个 next step，不要给五个模糊想法

研究沟通变好，靠的是变具体，不是变长。
